# Обзор проекта на русском

## Что это за проект
Это набор ботов для Clash of Clans.
В проекте есть:
- основной бот `v8` на `pyautogui` для фарма;
- Telegram-бот управления;
- отдельный `bot2` для другой деревни;
- отдельный `bot2_adb` для фоновой работы через ADB.

## Главные точки входа
- `bot.py` — основной цикл главного бота.
- `telegram_control_aiogram.py` — Telegram-панель и управление ботом.
- `bot2.py` — второй бот на `pyautogui`.
- `bot2_adb.py` — второй бот через ADB.

## Что делает основной бот
`bot.py` управляет всем циклом:
1. ждет стартовую задержку;
2. определяет, что он на базе;
3. запускает поиск атаки;
4. проверяет, что бой открыт;
5. читает золото и эликсир в бою;
6. если лута мало — скипает базу;
7. если лут подходит — высаживает войска, героев, спеллы;
8. сдается и возвращается домой;
9. обновляет счетчик атак;
10. периодически проверяет хранилища;
11. при включенной функции запускает проходку стен по таймеру.

## Блоки основного бота

### `attack_runtime/navigation.py`
Отвечает за:
- зум;
- выравнивание экрана;
- последовательность клавиш для поиска атаки.

### `attack_runtime/deploy.py`
Отвечает за:
- высадку машины;
- высадку героев;
- высадку войск;
- выпуск спеллов;
- прожатие способностей героев.

### `attack_runtime/spell_modes.py`
Определяет режимы спеллов:
- `stoneDick`
- `crazyWalls`

### `attack_runtime/surrender.py`
Отвечает за:
- ожидание до сдачи;
- нажатия для выхода из боя;
- проверку `go_home` перед возвратом.

### `attack_runtime/guards.py`
Отвечает за ожидание важных экранов во время атаки.
Сейчас главное применение — `go_home` после боя.

### `bot_runtime/screen_state.py`
Это общий модуль распознавания экранов.
Он умеет:
- собирать список шаблонов по маске файлов;
- искать картинки на экране;
- ждать `home`;
- ждать `battle ready`;
- вести счет попыток и отдавать причину затыка в recovery.

## OCR и лут

### `gold_filter.py`
Это модуль чтения ресурсов в бою.
Он:
- делает crop по золоту и эликсиру;
- гоняет OCR;
- использует fallback-регионы;
- умеет использовать шаблоны цифр как fallback;
- логирует, что именно увидел;
- решает, атаковать или скипать базу.

Это один из самых чувствительных модулей в проекте.
Если бот пропускает жирные базы, проблема чаще всего здесь:
- плохой регион;
- OCR-шум;
- неправильный crop;
- слабое чтение эликсира.

## Recovery

### `recovery_watchdog.py`
Это модуль устойчивости.
Он проверяет:
- черный экран;
- экраны ошибок по шаблонам;
- отсутствие прогресса;
- слишком долгий цикл;
- залипший экран;
- неожиданные исключения в основном цикле.

Если находит проблему, он:
- логирует ее;
- шлет уведомление в Telegram;
- перезапускает игру;
- возвращает бота в основной цикл.

Сейчас это один из самых полезных модулей проекта.

## Мониторинг хранилищ

### `storage_monitor.py`
Периодически проверяет золото и эликсир в хранилищах.
Если оба переполнены, бот:
- шлет сообщение в Telegram;
- останавливается.

## Переключение аккаунтов

### `account_switcher.py`
Сейчас переключение работает так:
1. бот возвращается домой;
2. открывает меню аккаунтов;
3. для известных аккаунтов кликает по координатам;
4. если нужно, использует fallback по шаблонам;
5. подтверждает, что снова на базе;
6. при необходимости нажимает `okay*.png`.

Это уже рабочий и достаточно практичный путь.
Лучший режим сейчас — координатный.

## Telegram-бот

### `telegram_control_aiogram.py`
Это центр управления.
Через него доступны:
- старт/стоп;
- скриншот;
- конфиг;
- спеллы;
- переключение аккаунтов;
- recovery debug;
- консольный режим;
- сценарий `Заполнить хранилища везде`.

### `telegram_ui/presentation.py`
Это сборка интерфейса:
- клавиатуры;
- меню;
- статусы;
- тексты панелей.

### `telegram_ui/process_controller.py`
Управляет процессом основного `bot.py`:
- запуск;
- остановка;
- PID;
- статус.

### `telegram_reporter.py`
Отдельный модуль отправки:
- текста в Telegram;
- скриншотов;
- attack log;
- console log.

## Bot2

### `bot2_runtime/*`
Это второй бот на `pyautogui`.
Подходит, когда можно использовать активный экран.

### `bot2_adb_runtime/*`
Это второй бот через ADB.
Подходит для фоновой работы без привязки к активному окну.
Это более правильное направление для параллельной работы.

## Что в проекте уже сильное
- основной цикл бота рабочий;
- recovery стал лучше и покрывает больше сбоев;
- Telegram-панель уже реально управляет проектом;
- координатное переключение аккаунтов работает лучше image-only подхода;
- у `bot2` и `bot2_adb` уже есть нормальная модульная структура.

## Что слабое
- OCR лута все еще нестабилен;
- Telegram-контроллер стал слишком большим;
- есть логика, завязанная на память процесса, а не на state;
- Windows-скрипты по качеству ниже macOS shell-скриптов;
- нет одного общего стартап-валидатора конфига и ресурсов.

## Что стоит улучшать дальше
1. Почистить `telegram_control_aiogram.py` и убрать дубли/лишнюю связанность.
2. Довести OCR лута до регрессионной проверки на реальных скринах.
3. Сделать startup-checker по конфигу, регионам, путям к картинкам и координатам.
4. Зафиксировать режимы запуска, чтобы два foreground-бота не мешали друг другу.
5. Постепенно переносить фоновые сценарии в ADB-first архитектуру.
